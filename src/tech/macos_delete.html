Case Study: delete confirmation in macOS

Since the earliest days of Macintosh OS (now macOS) Apple has placed emphasis on presenting users with a consistent interface and experience. The authoritative reference for designing consistent applications is the Human Interface Guidelines (HIG) published by Apple.

This page is a case study of one function that is common to many applications: deleting items or records. I will test the behaviour of various applications included with macOS and then classify them into categories. When I observe strange behaviours, I will add notes.

Well-Behaved and Fluid

The ideal user experience is created when the application supports the user in being safe, while also reducing friction by getting out of the way as much as possible. This ideal is exemplified by applications that delete items immediately without a prompt, while also making it easy to undo accidental deletions. All of the following applications fit this behaviour:

  • Keynote (delete slide)
  • Calendar (delete event)
  • Finder (delete file)
  • Mail (delete message)
  • Voice Memos (delete memo)
  • Notes (delete note)
  • Preview (delete page)
  • Reminders (delete reminder)
  • Safari (delete bookmark or bookmark folder)

Well-Behaved, but Mildly Annoying

Some applications add unnecessary friction to the process by displaying a modal sheet to confirm deletion. Well-behaved applications will use the same keyboard shortcuts to confirm or block the delete action. Applications should ideally still support undo, but may instead present warning text that the operation cannot be undone. A modal sheet provides helpful delay when thinking about irreversible operations.

No Undo History

Deletions in these applications cannot be un-done. The modal sheet should provide guidance that undo is not possible, but unfortunately it is not mentioned.

  • News (unfollow a source)
  • Keychain Access (delete an item)

Inconsistent key bindings

Each of these applications deviates from the normal keybindings in some way.

  • Messages (delete a conversation). The return key is bound to cancel (instead of delete). The only keybinding to confirm deletion is Cmd-D, which is not used anywhere else.
  • Stocks (delete stock). Return doesn't confirm deletion. Undo is unavailable, but not mentioned.
  • Home (delete a scene). Scenes cannot be selected with the keyboard. Deletion cannot be initiated with the keyboard. The confirmation prompt cannot be accepted or dismissed by the keyboard.
  • Podcasts (remove a podcast) There is no keybinding that will confirm deletion. The return key is bound to cancel instead of delete. The operation cannot be undone, and there is no warning about permanence. All other keyboard events fall through the modal -- for example, pressing the spacebar toggles the play/pause behaviour of the podcast in the parent window.

Other

  • Notes (delete a folder). The items in the folder are not put back into the folder when the folder is restored. Also, Notes crashed 10 seconds after restoring a deleted item.
